# 跨界网络仿真与物理桥接部署实战指南 (保姆级)

## 1. 核心设计理念与网络规划

本项目的核心目标是打破虚拟仿真环境（基于 OVS 和 Python 仿真器的 OSPF 网络）与物理宿主机（Windows）的“次元壁”。

* ### 1.1 架构思想：双网卡物理隔离 (极度重要)
  在我们的 Ubuntu 宿主机中，我们严格采用**双物理网卡**架构来分离流量，避免实验路由风暴导致宿主机失联：
  * **`ens37` (管理网 / 员工通道)**：VMware 分配的首张网卡（NAT 模式），负责为 Ubuntu 提供稳定的 SSH 远程连接通道。
  * **`ens33` (业务网 / 业务大门)**：VMware 新增的第二张专职桥接网卡（**必须强制设置为 NAT 模式以唤醒底层驱动**），专职用于桥接内部的 OSPF 网络与外部 Windows。井水不犯河水。 

### 1.2 全网 IP 规划说明
| 网络区域         | 网段 (Subnet)      | 核心设备 IP                           | 作用说明                                                    |
| :--------------- | :----------------- | :------------------------------------ | :---------------------------------------------------------- |
| **PC1 终端网段** | `10.1.1.0/24`      | PC1: `10.1.1.2`, R1网关: `10.1.1.1`   | 模拟内部最左侧的终端业务网络                                |
| **骨干互联 1**   | `20.1.1.0/24`      | R1: `20.1.1.2`, R3: `20.1.1.1`        | 路由器 R1 与 R3 之间的 P2P 互联                             |
| **骨干互联 2**   | `21.1.1.0/24`      | R3: `21.1.1.1`, R2: `21.1.1.2`        | 路由器 R3 与 R2 之间的 P2P 互联                             |
| **PC2 终端网段** | `30.1.1.0/24`      | PC2: `30.1.1.2`, R2网关: `30.1.1.1`   | 模拟内部最右侧的终端业务网络                                |
| **跨界桥接网段** | `192.168.153.0/24` | Windows: `153.1`, R1跨界口: `153.200` | **打破次元壁的核心！** 必须与 VMware 的 VMnet8 保持同一网段 |

## 2. 准备工作：VMware 虚拟机底层配置

在启动任何脚本之前，必须在 Windows 和 VMware 层面打好物理基础。

### 2.1 添加专属桥接网卡 (`ens33`)
1. 关闭 Ubuntu 虚拟机。
2. 右键虚拟机 -> **设置 (Settings)** -> 点击底部的 **添加 (Add)**。
3. 选择 **网络适配器 (Network Adapter)**，点击完成。
4. 将新加的网卡（通常是“网络适配器 2”）模式设置为 **NAT 模式**（对应 VMnet8）。
5. 确保勾选了 **“启动时连接”**。开机进入 Ubuntu。

### 2.2 确认宿主机网段 IP
1. 在 Windows 宿主机搜索并打开 **“虚拟网络编辑器 (Virtual Network Editor)”**。
2. 找到 **VMnet8 (NAT模式)**，在下方查看其“子网 IP”。假设为 `192.168.153.0`。
3. 在 Windows 宿主机打开 CMD，输入 `ipconfig`，确认 `VMware Network Adapter VMnet8` 的 IPv4 地址（通常为 `192.168.153.1`）。**这个 IP 就是 R1 最终要 Ping 通的目标！**

## 3. 仿真引擎代码改造 (接入 Cloud 节点)  无视

### 3.1 拓扑文件 `topology.yaml` 改造
在拓扑的末尾新增 `cloud` 节点，并在 `links` 中绑定刚刚添加的专属网卡 `ens33`。
```yaml
  # --- 新增跨界云节点 ---
  - name: CLOUD
    type: cloud

links:
  # 将边界路由器 R1 的第3个接口 (下标为3) 映射到物理桥接网卡 ens33
  - ["R1", 3, "CLOUD", "ens33"]
```

### 3.2 引擎核心 `main.py` 改造 (防报错逻辑)

Python

```
# 1. 防止云节点缺少控制台端口导致 KeyError:
port = dev_conf.get('console_port')

# 2. 拦截对云节点的无意义启动导致 AttributeError:
if device and device != "CLOUD_NODE":
    device.start()
```

## 4. 全网贯通实施步骤 (Ctrl+C/V 直接起飞)

### 步骤一：底座清理与拓扑拉起

在 Ubuntu (通过 ens37 SSH 登录) 打开第一个终端：

Bash

```
# 清空可能残留的虚拟进程和旧配置镜像
python3 main.py clean

# 拉起全新拓扑
python3 main.py load topology.yaml
```

*保持此终端运行，观察是否输出“成功桥接 ens33”等字样。*

### 步骤二：开启网卡混杂模式 (极度重要)

新开一个 Ubuntu 终端。为了绕过 VMware 虚拟交换机的 MAC 地址过滤机制，必须开启混杂模式（Promiscuous Mode）。

Bash

```
# 强行唤醒网卡
sudo ip link set ens33 up
# 开启物理网卡混杂模式
sudo ip link set ens33 promisc on
# 开启 OVS 虚拟网桥混杂模式
sudo ip link set br_R1_ge3 promisc on 
```

### 步骤三：配置底层 PC 节点

新开终端，配置两台 Linux 模拟 PC 的 IP 和默认网关。

Bash

```
# 登录 PC1 (telnet 127.0.0.1 5004)
sudo su
ip addr add 10.1.1.2/24 dev eth0
ip link set eth0 up
ip route add default via 10.1.1.1

# 登录 PC2 (telnet 127.0.0.1 5005)
sudo su
ip addr add 30.1.1.2/24 dev eth0
ip link set eth0 up
ip route add default via 30.1.1.1
```

### 步骤四：配置二层接入交换机 (H3C)

将连接 PC 的端口划入 VLAN。

Plaintext

```
# 登录 S1 (telnet 127.0.0.1 6001)
system-view
vlan 10
quit
interface range HGE1/0/1 to HGE1/0/2
port link-mode bridge
port link-type access
port access vlan 10
undo shutdown
quit
return

# 登录 S2 (telnet 127.0.0.1 6002)
system-view
vlan 30
quit
interface range HGE1/0/1 to HGE1/0/2
port link-mode bridge
port link-type access
port access vlan 30
undo shutdown
quit
return
```

### 步骤五：配置核心路由器与 OSPF (华为 VRP)

配置内部 OSPF 路由，并为边界路由器 R1 配置与 Windows VMnet8 同网段的跨界 IP (`192.168.153.200`)。

Plaintext

```
# 登录 R1 (边界路由器 - telnet 127.0.0.1 5001)
system-view
interface Ethernet1/0/0
ip address 10.1.1.1 24
commit
quit
interface Ethernet1/0/1
ip address 20.1.1.2 24
commit
quit
interface Ethernet1/0/2
ip address 192.168.153.200 24  # 跨界大门 IP
commit
quit
ospf 1
area 0
network 10.1.1.0 0.0.0.255
network 20.1.1.0 0.0.0.255
network 192.168.153.0 0.0.0.255 # 将跨界网段宣告进 OSPF
commit
return

# 登录 R3 (骨干路由器 - telnet 127.0.0.1 5003)
system-view
interface Ethernet1/0/0
ip address 20.1.1.1 24
commit
quit
interface Ethernet1/0/1
ip address 21.1.1.1 24
commit
quit
ospf 1
area 0
network 20.1.1.0 0.0.0.255
quit
area 1
network 21.1.1.0 0.0.0.255
commit
return

# 登录 R2 (内部路由器 - telnet 127.0.0.1 5002)
system-view
interface Ethernet1/0/0
ip address 30.1.1.1 24
commit
quit
interface Ethernet1/0/1
ip address 21.1.1.2 24
commit
quit
ospf 1
area 1
network 30.1.1.0 0.0.0.255
network 21.1.1.0 0.0.0.255
commit
return
```

### 步骤六：宿主机 (Windows) 静态路由注入

在 Windows 本机搜索 `cmd`，**右键选择“以管理员身份运行”**。将去往 10/20/30 网段的数据包，强行指向虚拟路由器 R1。

DOS

```
route add 10.1.1.0 mask 255.255.255.0 192.168.153.200
route add 20.1.1.0 mask 255.255.255.0 192.168.153.200
route add 30.1.1.0 mask 255.255.255.0 192.168.153.200
```

### 步骤七：终极连通性验证

1. **测试内部 OSPF**：在 PC1 (5004) 执行 `ping 30.1.1.2`，通则说明 OSPF 与 VLAN 完美收敛。
2. **测试边界桥接**：在 R1 (5001) 执行 `ping 192.168.153.1`，通则说明底层 OVS 与物理网卡打通。
3. **测试跨界全网**：在 Windows CMD 执行 `ping 10.1.1.2`，通则说明三层路由穿越物理机与虚拟机，实验大获全胜！

## 5. 🚨 避坑与故障排查指南 (Troubleshooting)

### 坑 1：100% 丢包，连边缘路由器大门都敲不开

- **现象**：Windows `ping 192.168.153.200` 请求超时。
- **排查**：在 R1 中执行 `display arp all`。若无宿主机 MAC 地址，说明二层未通；在 Ubuntu 用 `sudo tcpdump -i ens33 -n arp or icmp` 抓包看请求卡在哪里。
- **解法**：**必须关闭 Windows Defender 防火墙**（Windows 默认静默丢弃非本地子网的 ICMP 报文）。确认 Ubuntu 的 `promisc` 混杂模式已开启。

- 

### 坑 2：最绝望的底层死锁 (NO-CARRIER) 与终极逃生方案

* **现象**：在 Ubuntu 执行 `ip link show ens33` 时，网卡显示 `state DOWN` 且伴随 `<NO-CARRIER>` 标志。此时内核拒绝发送任何数据包，表现为 100% 丢包。
* **原因**：VMware 的多网卡环路保护机制触发，或者虚拟网卡底层驱动卡死，未能向该网口注入载波电信号。
* **解法一（实战首选：切换 NAT 模式唤醒）**：
  打开 VMware 虚拟机设置，找到 `ens33` 对应的网络适配器。**无论它当前是什么模式，强制将其修改为【NAT 模式】并保存**。这个动作会强制 VMware 重新初始化该网卡的底层物理驱动，瞬间消除 `NO-CARRIER`。
* **解法二（常规物理唤醒）**：在 VMware 右下角网卡图标“断开连接”再“重新连接”；或在 Windows 宿主机重启 VMware NAT Service 服务。
* **解法三（实战逃生通道 - 借道管理网）**：若专属网卡 `ens33` 彻底损坏无法唤醒，可紧急将 OVS 网桥强行挂载到负责 SSH 的健康主网卡 (`ens37`) 上以完成实验验收：
  `sudo ovs-vsctl del-port br_R1_ge3 ens33`
  `sudo ovs-vsctl add-port br_R1_ge3 ens37`
  *(注：此举为非标避险方案，可能导致管理网卡流量混杂，用完后建议尽早恢复)*
